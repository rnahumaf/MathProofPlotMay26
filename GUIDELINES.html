Base

Princípios de trabalho

Preferência por execução pragmática, mudanças cirúrgicas e validação proporcional ao risco.

Como agir

  • Comece lendo o repositório; não presuma arquitetura, scripts ou convenções.
  • Explique suposições quando houver ambiguidade relevante.
  • Faça mudanças diretamente ligadas ao pedido.
  • Evite abstrair, generalizar ou adicionar flexibilidade sem demanda real.
  • Verifique o resultado e informe checks que não puder executar.

Progresso prático com critério de sucesso claro, sem transformar tarefas pequenas em processos grandes.

Base

Precisão editorial

Conteúdo deve ser compacto, direto e útil para execução.

Preferências

  • Prefira conteúdo compacto, direto e acionável.
  • Evite menções, citações e referências externas quando elas não melhorarem a instrução.
  • Registre preferências como regras práticas, não como justificativas longas.
  • Use exemplos apenas quando reduzirem ambiguidade real.

Base

Idioma e UTF-8

Projetos devem preservar PT-BR com acentuação correta e renderização UTF-8.

Regras

  • Use PT-BR com acentuação correta em textos, documentação e interfaces.
  • Arquivos HTML devem declarar <meta charset="UTF-8" />.
  • Preserve UTF-8 nos arquivos do projeto; não remova acentos por conveniência técnica.
  • Ao gerar conteúdo, confirme suporte a caracteres como á, é, í, ó, ú, ç e ã.

Repositório

AGENTS.md

Contrato de agente deve ser curto, mandatário e apontar para contexto sob demanda.

Organização

  • Mantenha o arquivo curto, mandatário e fácil de escanear.
  • Registre apenas regras globais, scripts essenciais, gotchas críticos e caminhos das skills.
  • Não cole arquivos fonte inteiros dentro do contrato.
  • Quando um detalhe muda com frequência, aponte para o arquivo real em vez de duplicar conteúdo.
  • Separe conhecimento reutilizável em skills específicas.

Repositório

Skills

Skills devem funcionar como memória operacional, não como documentação genérica.

Estrutura

  • Uma skill deve ser uma pasta, não apenas um markdown solto.
  • A descrição é o gatilho. Escreva quando usar, em termos concretos.
  • O corpo deve ser enxuto e operacional.
  • Use referências, scripts e assets como disclosure progressivo.
  • Inclua uma seção Gotchas com falhas reais e recorrentes.

Evite railroading: não gaste tokens dizendo o óbvio. Uma boa skill contém conhecimento local, não um tutorial genérico.

Repositório

README

README deve orientar uso real do projeto, sem duplicar contratos de agente.

Conteúdo esperado

  • Explique o que o projeto faz, como rodar, como testar e como publicar.
  • Priorize comandos reais do repositório sobre explicações longas.
  • Inclua dependências externas, variáveis de ambiente e dados necessários para desenvolvimento.
  • Evite duplicar regras de agente no README; use links para AGENTS.md e skills.
  • Atualize quando o fluxo de build ou deploy mudar.

Interface

HTML e CSS

Visual claro, denso, escaneável e inspirado em shadcn sem depender de framework.

Preferências visuais

  • Use tema claro por padrão: fundos claros, texto escuro e bordas sutis.
  • Prefira barras superiores discretas, compactas e com título proporcional ao restante da interface.
  • Para documentação, prefira sidebar esquerda colapsável e conteúdo principal por seção.
  • Evite cards como estrutura principal. Separe áreas com linhas suaves, retas e espaçamento consistente.
  • Mantenha raio de borda em torno de 8px ou menos, salvo padrão existente.
  • Não dependa de fundos escuros globais, gradientes pesados ou paletas de uma cor só.
  • Garanta responsividade sem texto sobreposto ou controles instáveis.

Interface

JavaScript

Use JS simples e previsível para comportamento de interface.

Regras

  • Use ES Modules quando o projeto permitir.
  • Não use handlers inline como onclick; registre eventos no JS.
  • Prefira APIs estruturadas a manipulação frágil de strings.
  • Adicione comentários apenas quando eles evitarem uma leitura trabalhosa de lógica não óbvia.
  • Remova imports, variáveis e funções que a sua própria mudança tornou inutilizados.

Qualidade

Gotchas e memória

Falhas recorrentes devem virar memória curta, concreta e acionável.

Registro

  • Quando um agente ou humano repetir o mesmo erro, registre o caso na skill correta.
  • Gotchas devem ser concretos: comando, ambiente, arquivo, limite, exceção ou comportamento observado.
  • Remova conteúdo antigo que deixou de refletir o repositório.
  • Prefira apontar para logs, scripts ou referências locais quando eles forem mais confiáveis que memória textual.

Qualidade

Verificação

A verificação deve ser proporcional ao risco e explícita no fechamento.

Checks

  • Defina sucesso antes de implementar quando a tarefa tiver risco ou vários passos.
  • Use o menor teste suficiente para a mudança.
  • Não faça validação visual, screenshots ou inspeção de UI renderizada após implementações, salvo pedido explícito do usuário; isso gasta muitos tokens e costuma ser ineficiente.
  • Não abra arquivos de imagem ou SVG sem autorização explícita do usuário. Esses arquivos podem ser muito grandes e consumir contexto rapidamente; inspeções visuais costumam ser feitas com mais eficiência pelo usuário.
  • Rode build completo quando tocar configuração, assets compartilhados, dados gerados ou muitas páginas.
  • Confira diffs após comandos geradores para separar mudanças intencionais de efeitos colaterais.
  • Declare qualquer verificação que ficou pendente.

Qualidade

Segurança e terceiros

Dependências e automações externas exigem leitura antes de confiança.

Cuidados

  • Audite skills, scripts e dependências de terceiros antes de confiar neles.
  • Procure chamadas de rede, escrita em disco, execução de shell e credenciais.
  • Não use flags destrutivas ou comandos de limpeza ampla sem necessidade explícita.
  • Preserve alterações do usuário em worktrees sujos.
